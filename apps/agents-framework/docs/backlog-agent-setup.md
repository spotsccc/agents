# Настройка агента для работы с backlog

## Цель

Этот документ фиксирует целевую техническую модель для `agents-framework`, в которой агент работает не напрямую из чата, а через backlog-артефакты в репозитории. Backlog становится единственным источником правды для планирования, анализа готовности и исполнения задач.

## Общая концепция

- Пользователь взаимодействует с агентом через backlog, а не через разрозненные одноразовые инструкции.
- Агент использует skills как операционные протоколы для отдельных сценариев работы с backlog.
- Исходная идея фичи или проекта сначала превращается в research-артефакт, затем в backlog-элементы, затем в готовые к исполнению work items.
- Backlog хранится в репозитории в человекочитаемом Markdown-формате, но при этом имеет строгую структуру, достаточную для работы AI coding agent.
- Backlog должен поддерживать иерархию, зависимости, порядок исполнения, приоритет и ссылки между элементами.
- Каноническое состояние backlog должно жить в файлах, а не в памяти агента или только в истории чата.

## Что должен настраивать `agents-framework`

Следующая итерация `agents-framework` должна инициализировать не только базовый backlog, но и полный минимальный набор артефактов для backlog-driven работы:

- агентный профиль в `.agents/.codex` или `.agents/.claude`;
- каталог общих backlog-данных в `.agents/backlog`;
- шаблоны для research и для каждого типа backlog-элемента;
- skills, покрывающие 4 основных сценария работы с backlog;
- managed-блок в `AGENTS.md`, который фиксирует обязательный flow работы через backlog.

## Целевая структура каталогов

Текущий MVP уже создает `.agents/backlog` и базовые skills. Для backlog с иерархией и несколькими сценариями работы целевая структура должна быть расширена:

```text
.agents/
  backlog/
    README.md
    inbox/
    research/
    items/
      projects/
      epics/
      tasks/
      subtasks/
    templates/
      research-template.md
      project-template.md
      epic-template.md
      task-template.md
      subtask-template.md
      readiness-checklist.md
    views/
      ready.md
      in-progress.md
      done.md
  .codex/
    README.md
    skills/
      backlog-workflow/
      backlog-research/
      backlog-planning/
      backlog-readiness/
      backlog-execution/
      engineering-standards/
      testing-verification/
```

## Каноническая модель backlog

Backlog должен быть иерархичным и состоять из 4 типов элементов:

- `project`
- `epic`
- `task`
- `subtask`

Иерархия должна быть строгой:

- `project` содержит `epic`
- `epic` содержит `task`
- `task` содержит `subtask`

Каждый backlog-элемент должен быть отдельным Markdown-файлом. Канонический источник данных должен храниться в `items/`, а статусные представления вроде `ready`, `in-progress` и `done` должны быть либо отдельными индексами, либо генерируемыми views. Это позволит одновременно сохранить иерархию и поддержать операционный flow.

## Структура backlog-элемента

Чтобы backlog был одновременно человекочитаемым и пригодным для исполнения агентом, каждый элемент должен состоять из двух частей:

- структурированные метаданные в YAML frontmatter;
- фиксированные Markdown-секции в теле документа.

Минимальный набор обязательных метаданных:

- `id` — стабильный идентификатор;
- `type` — один из `project`, `epic`, `task`, `subtask`;
- `title` — короткое человекочитаемое название;
- `status` — например `draft`, `ready`, `in-progress`, `done`, `blocked`;
- `priority` — относительный приоритет;
- `order` — позиция внутри родительского элемента;
- `parent` — ссылка на родительский backlog-элемент;
- `depends_on` — список зависимостей;
- `blocks` — список элементов, которые блокируются текущим;
- `research` — ссылки на research-документы;
- `updated_at` — дата последнего существенного изменения.

Минимальный набор секций в теле документа:

- `Summary`
- `Scope`
- `Acceptance Criteria`
- `Implementation Notes`
- `Verification Plan`
- `Links`
- `Decision Log`

Пример канонического формата:

```md
---
id: AF-TASK-017
type: task
title: Add backlog readiness evaluation
status: ready
priority: high
order: 30
parent: AF-EPIC-003
depends_on:
  - AF-TASK-012
blocks: []
research:
  - ../research/AF-RSCH-005.md
updated_at: 2026-03-13
---

# AF-TASK-017 Add backlog readiness evaluation

## Summary

Краткое описание задачи и ожидаемого результата.

## Scope

- In scope:
- Out of scope:

- [ ]

## Implementation Notes

- Affected files:
- Constraints:

## Verification Plan

- Automated checks:
- Manual checks:

## Links

- Parent:
- Research:

```

## Основные сценарии работы агента

### 1. Проведение research

Вход:

- идея фичи;
- идея проекта;
- плохо сформулированный запрос на изменение.

Действия агента:

- проходит по коду и структуре проекта;
- находит релевантные файлы, модули, точки интеграции и тесты;
- фиксирует предполагаемые изменения, ограничения, зависимости и риски;
- формирует предварительный план реализации и верификации.
- Делает ресерч того, как такую фичу можно реализовать в текущем проекта
- Какие есть ограничения технические и бизнесовые

Результат:

- research-файл в `.agents/backlog/research/`;
- список затрагиваемых частей проекта;
- предварительный тестовый план;
- список открытых вопросов и предположений.

### 2. Создание backlog-элементов из research

Вход:

- готовый research-документ;
- при необходимости ответы пользователя на уточняющие вопросы.

Действия агента:

- анализирует research;
- выделяет отдельные deliverables;
- разбивает работу на `project`, `epic`, `task`, `subtask`;
- расставляет порядок, приоритет, родительские связи и зависимости;
- создает backlog-элементы в каноническом формате.

Результат:

- набор связанных backlog-файлов;
- ссылки на research;
- явный execution order;
- backlog, пригодный для исполнения агентами без скрытого контекста.

### 3. Анализ backlog-элемента на готовность к исполнению

Вход:

- один backlog-элемент.

Агент должен проверить:

- достаточно ли четко описан scope;
- есть ли измеримые acceptance criteria;
- понятны ли зависимости и блокеры;
- есть ли ссылки на research или на родительский контекст;
- перечислены ли релевантные файлы, системы и тесты;
- есть ли план верификации;
- остались ли критические неизвестные, которые мешают исполнению.

Результат:

- оценка готовности;
- список недостающей информации;
- рекомендации, что нужно уточнить до перевода элемента в `ready`.

### 4. Передача элемента на исполнение

Предусловия:

- backlog-элемент имеет статус `ready`;
- у него нет неразрешенных блокирующих зависимостей;
- для него есть достаточный контекст для кодинга и тестирования.

Действия агента:

- берет элемент в работу;
- переводит его в `in-progress`;
- использует связанные skills для реализации;
- обновляет `Decision Log`, `Implementation Notes` и `Verification Plan` по ходу работы;
- после завершения переводит элемент в `done`.

Результат:

- исполнение происходит по явному backlog-контракту;
- принятые решения и проверки остаются в репозитории;
- следующий агент может продолжить работу без потери контекста.

## Требования к backlog

Система backlog в `agents-framework` должна удовлетворять следующим требованиям:

1. Backlog иерархичный.
2. Элементы имеют строгую структуру.
3. Элементы имеют порядок и приоритет.
4. Элементы могут быть связаны и ссылаться друг на друга.
5. Поддерживаются 4 типа элементов: проект, эпик, задача, подзадача.
6. Элементы читаемы для человека.
7. Элементы пригодны для исполнения AI coding agent.

## Верхнеуровневый набор skills

Для реализации такого поведения нужен не один общий skill, а набор специализированных skills.

### Базовые backlog-skills

- `backlog-workflow` — общий контракт работы через backlog, правила смены статусов и ведения task history.
- `backlog-research` — протокол проведения research по идее, поиска релевантных файлов, кода, тестов и рисков.
- `backlog-planning` — преобразование research в иерархию backlog-элементов, разбиение на deliverables, приоритизацию и связывание зависимостей.
- `backlog-readiness` — анализ готовности backlog-элемента к исполнению и фиксация недостающего контекста.
- `backlog-execution` — правила взятия элемента в работу, обновления статуса, ведения decision log и завершения задачи.

### Поддерживающие skills

- `engineering-standards` — общие правила по качеству кода, документации и размеру изменений.
- `testing-verification` — обязательный подход к тестовому плану, обновлению тестов и фиксации manual checks.
- технологические skills, зависящие от проекта, например `javascript`, `sql`, `playwright`, `vitest`, `vercel`.

## Связь с текущим MVP

Текущий `agents-framework` уже содержит правильный фундамент:

- создает `.agents/backlog`;
- создает профиль агента и каталог skills;
- обновляет `AGENTS.md`;
- задает backlog-driven подход как основной flow.

Следующий шаг развития MVP:

- расширить backlog с плоского task-flow до канонической иерархии `project -> epic -> task -> subtask`;
- добавить research как отдельный артефакт первого класса;
- разделить текущий общий backlog skill на scenario-specific skills;
- расширить шаблоны backlog-элементов, чтобы они были пригодны и для человека, и для AI agent.
